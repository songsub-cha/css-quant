# 0001. UUID v7을 기본 키(PK)로 사용

- **날짜**: 2026-07-11
- **상태**: Accepted
- **관련**: SoT [PART E1](../SoT.md#e1-adr-새-저장소-docsadr에-재기록) #1, [B4.1](../SoT.md#b4-표준)

## 맥락

새 저장소는 구 저장소(`quant_ai`) PRD v7의 검증된 결정을 승계한다. 모든 테이블의 기본 키 타입을 하나로 통일해야
인덱스 전략, ID 생성 방식, 외부 노출 ID 포맷을 일관되게 설계할 수 있다. 순차 증가 정수 PK는 다중 서비스/워커
환경에서 충돌 없는 사전 생성이 어렵고, 무작위 UUID v4는 인덱스 지역성이 나빠 대량 쓰기 워크로드(가격·팩터·
점수 등 일별 배치)에서 성능이 저하된다.

## 결정

모든 테이블의 PK는 **UUID v7**을 사용한다. 외부에 노출되는 ID는 리소스 유형을 식별할 수 있도록 prefix를
붙인다 (`usr_`, `str_`, `bt_`, `ord_`, `oc_`, `pf_` 등).

## 결과

- UUID v7은 타임스탬프 기반이라 **시간 정렬이 가능**하며, B-tree 인덱스 지역성이 UUID v4보다 우수해
  일별 배치로 대량 삽입되는 `market_prices`, `asset_factors`, `ai_scores` 등에 유리하다.
- 모든 테이블은 `id UUID PK` + `created_at/updated_at timestamptz` 컬럼을 갖는 표준 패턴을 따른다
  (SoT C3 ER 개요).
- prefix 규칙 덕분에 로그·에러 응답·API 경로에서 ID만 보고도 리소스 종류를 식별할 수 있다.
