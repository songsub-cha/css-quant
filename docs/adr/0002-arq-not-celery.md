# 0002. Arq 채택 (Celery 대신)

- **날짜**: 2026-07-11
- **상태**: Accepted
- **관련**: SoT [PART E1](../SoT.md#e1-adr-새-저장소-docsadr에-재기록) #2, [D6](../SoT.md#d6-배치-스케줄-arq-크론-utc-정의--kst-병기--구-구현-검증값), [B5.4](../SoT.md#b5-검증된-구현-결정-구-저장소에서-확인된-함정--반드시-승계)

## 맥락

일일 파이프라인(가격 수집 → 품질 게이트 → 팩터 → 점수 → LLM 설명 → 전략 실행), 백테스트 비동기 실행,
공시 감시, 알림 발송 등 다수의 백그라운드 잡·크론이 필요하다. 백엔드 프레임워크(FastAPI)는 async 네이티브로
설계되며, 인프라는 Postgres + Redis로 최소화한다는 원칙(B1)을 따른다.

## 결정

작업 큐/크론 엔진으로 **Arq**를 채택한다 (Celery는 사용하지 않는다).

## 결과

- Arq는 asyncio 네이티브라 FastAPI 서비스 레이어의 async 코드를 워커에서 그대로 재사용할 수 있다.
- 의존 인프라가 **Redis 하나**로 끝난다 — Celery처럼 별도 브로커/결과 백엔드 조합을 구성할 필요가 없다.
- 단, Arq는 Redis 가용성에 강하게 의존한다. 다른 Redis 사용처(rate limit, 캐시)는 연결 실패 시 허용적
  기본값으로 우아하게 성능 저하하지만, **워커(Arq) 관점에서 Redis는 필수 가용성 대상**이므로 healthcheck와
  restart 정책을 갖춘다 (B5.4).
- D6의 모든 배치 스케줄(크론)과 C4의 `job_runs` 기록은 Arq 태스크로 구현된다.
- `worker` 컨테이너는 prod Docker Compose에 반드시 포함되어야 한다 — 누락되면 파이프라인 전체가 조용히
  돌지 않는 사고로 이어진다 (D1, 구 구현에서 실증된 함정).
